# sync-skills 用户故事

本文档记录 `sync-skills` 在当前定位下的用户故事。

当前定位不是“下载别人 Skill 的工具”，而是：

> 自建 Skill 的全生命周期管理器

因此本文件优先覆盖：

- 创建
- 纳管
- 迭代
- 本地分发
- 远程备份 / 分享
- 多机同步
- 删除 / 下线

而不把“公网搜索和安装别人 Skill”作为主故事。

---

## 1. 状态说明

| 状态 | 含义 |
|------|------|
| `已实现` | 当前代码和测试已经覆盖 |
| `部分实现` | 当前已有能力，但体验、边界或入口还不完整 |
| `待实现` | 当前还没有完整支持，但属于合理扩展 |
| `非目标` | 当前定位下不作为主目标推进 |

## Roadmap

本节只记录当前主线的后续故事。凡是单 Skill 包发布、第三方 Skill 搜索或 registry 化能力，均不属于本阶段 roadmap。
Roadmap 暂不单独拆文档：`docs/DESIGN.md` 维护产品级路线，`docs/USER_STORIES.md` 维护故事级路线，两者一起表达当前最新状态。
其中 `US-RM-1` 到 `US-RM-3` 已实现，保留在此作为里程碑记录，避免 roadmap 状态滞后。

### US-RM-1 `doctor --dry-run` 真正只读

状态：`已实现`
阶段：诊断与恢复
故事：作为维护者，我希望在执行 `doctor --dry-run` 时只看到拟执行修复，而不会改动任何文件系统状态或配置状态，这样我可以先评估风险，再决定是否真正修复。
说明：当前实现已保证只输出拟执行动作、跳过原因和无操作结果，不改写 state、symlink 或目录结构。
验收标准：
1. 不创建、删除或改写 symlink、目录、仓库文件和 state 文件。
2. 输出包含拟执行动作、跳过原因和无操作结果。
3. 没有待修复项时直接退出，不要求额外确认。

### US-RM-2 `status` 生命周期状态补全

状态：`已实现`
阶段：日常管理
故事：作为维护者，我希望 `status` 能直接告诉我每个 Skill 当前是健康、断链、冲突、残留还是未分发，这样我不需要再用多个命令推断系统状态。
说明：当前实现已显式区分 `managed`、`unknown`、`broken link`、`real directory conflict`、`orphaned`、`managed but not exposed` 等状态。
验收标准：
1. 能显式区分 `managed`、`unknown`、`broken link`、`real directory conflict`、`orphaned`、`managed but not exposed` 等状态。
2. 状态命名和 `doctor` 的修复动作保持一致。
3. 用户无需再依赖手工检查 Agent 目录推断状态。

### US-RM-3 orphaned 与残留 state 收口

状态：`已实现`
阶段：诊断与恢复
故事：作为维护者，我希望系统能识别并清理 orphaned 和残留 state，这样长期使用后管理状态不会越来越脏。
说明：当前实现中，`status` 已能暴露这类异常，`doctor` 在真实执行时会清理 orphaned state。
验收标准：
1. `status` 能显式暴露这类异常。
2. `doctor` 能给出明确修复动作。
3. 修复后 state 与文件系统状态重新一致。

### US-RM-4 Git 异常恢复提示

状态：`已实现`
阶段：协作与同步
故事：作为维护者，我希望在 `commit`、`push`、`pull` 失败时获得准确、可执行的提示，这样我可以安全处理远程、分叉、冲突和空提交等情况。
说明：当前实现已覆盖 `git` 不可用、首次 upstream 建立、远程未配置、无改动提交、本地领先或落后、与远程分叉、认证失败、detached HEAD、`pull` 冲突、本地未提交改动和缺失远程分支等场景，并在失败时给出明确的下一步提示。
验收标准：
1. 能识别远程未配置、首次 upstream、无改动提交、本地领先或落后、分叉和冲突等常见场景。
2. 提示信息说明当前状态、为什么失败、下一步该怎么做。
3. 失败时不发生额外的隐式修改。

### US-RM-5 现有仓库接入与多机恢复

状态：`部分实现`
阶段：接入与初始化
故事：作为在新电脑上继续使用 Skill 仓库的维护者，我希望能够用现有仓库快速恢复本地 state 和 Agent 可见性，而不需要大量手工操作。
验收标准：
1. `init`、`pull`、`doctor` 形成清晰恢复主线。
2. 文档能够明确说明何时需要 `pull`，何时需要 `doctor`。
3. 常见恢复失败场景有明确提示。

### US-RM-6 多 Agent 目标管理可见性

状态：`待实现`
阶段：日常管理
故事：作为同时服务多个 Agent 的维护者，我希望清楚知道每个 Skill 已经暴露给哪些 Agent、哪些 Agent 还未暴露、哪些目标路径存在问题，这样我能稳定管理本地分发。
验收标准：
1. `status` 或相关输出能展示 Agent 维度的可见性结果。
2. `link` 和 `unlink` 的输出能明确说明受影响目标。
3. 目标路径不可写或不存在时，错误信息足够明确。

### US-RM-7 统一 v1 与 legacy 查询入口

状态：`待实现`
阶段：CLI 一致性
故事：作为维护者，我希望 v1 模式和 legacy 模式的 `list`、`search`、`info` 入口边界清晰一致，这样我不会因为命令被错误路由而得到意外结果。
验收标准：
1. `list`、`search`、`info` 的行为边界明确，不再依赖隐式竞争。
2. 文档、CLI 路由和测试对命令归属保持一致。
3. 用户能明确知道哪些命令属于当前主线，哪些仍属于 legacy。
